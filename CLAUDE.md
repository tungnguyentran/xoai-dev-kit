## Overview

XoaiUtility is a multi-platform SwiftUI app (macOS / iOS / visionOS, deployment target 26.5, Swift 5.0). Despite the SwiftData scaffolding generated by the Xcode template, its real feature is **DevKit**, a multi-tool developer utility: a left sidebar selects between four tools — **JSON Formatter** (`JsonTool`), **URL encode/decode** (`UrlTool`), **Base64** (`Base64Tool`), and **JWT decode** (`JwtTool`) — each with an input-on-top / output-below layout, a shared right-hand **history** panel, and a Dark/Light theme toggle. The UI targets macOS (uses `NSPasteboard`, `AppKit`, oklch colors). `ContentView` injects three `EnvironmentObject`s — `ThemeManager`, `AppModel`, `LocalizationManager` — into `RootView`.

The entire UI is **localized VI / English** (see Localization below).

## Build & Test

There is no `.xcworkspace` for a Swift package — use the Xcode project. The scheme is `XoaiUtility`.

```bash
# Build (pick a destination matching SUPPORTED_PLATFORMS: macosx, iphonesimulator, xrsimulator)
xcodebuild -project XoaiUtility.xcodeproj -scheme XoaiUtility -destination 'platform=macOS' build

# Run all tests (unit + UI)
xcodebuild -project XoaiUtility.xcodeproj -scheme XoaiUtility -destination 'platform=macOS' test

# Run a single test (Swift Testing, not XCTest, for unit tests)
xcodebuild test -project XoaiUtility.xcodeproj -scheme XoaiUtility -destination 'platform=macOS' \
  -only-testing:XoaiUtilityTests/XoaiUtilityTests/example
```

Unit tests (`XoaiUtilityTests`) use the **Swift Testing** framework (`import Testing`, `@Test`, `#expect`) — not XCTest, and cover the JSON engine, oklch conversion, `AppLog`, and localization. UI tests (`XoaiUtilityUITests`) are XCTest template scaffolding; they **fail in non-interactive sessions** ("Failed to activate application … Running Background") because the runner can't foreground the GUI app — this is environmental, not a code failure. Gauge correctness from `XoaiUtilityTests`.

## Architecture

The app is split into focused files (no single mega-file):

- **State managers** (injected as `EnvironmentObject` in `ContentView`):
  - `AppModel` — active tool, history (persisted to `UserDefaults` key `devkit-history`), and a `Seed` mechanism for reloading a history entry into a tool's input. `ToolID` is the tool enum; its `name` is a **stable, non-localized** os.Logger category (used by `AppLog`), while `displayName/displayDesc/displayShortName(_ s: Strings)` give localized UI text.
  - `ThemeManager` (`Theme.swift`) — `@AppStorage("devkit-theme")`; exposes `theme.t` (a `ThemeTokens` color set authored in **oklch**, converted to sRGB at runtime via `Color(oklch:…)`).
  - `LocalizationManager` (`Localization.swift`) — see Localization below.

- **Shell & components:** `RootView` (sidebar | header + tool + history), `Sidebar` (tool list + VI/EN and Dark/Light toggles), `HistoryPanel`, and `DevKitComponents.swift` (shared `Btn`, `Pane`, `Segmented`, `CopyBtn`, `CountBar`, `CodeArea`, `Banner`, etc.).

- **Tools:** each `*Tool.swift` pairs a pure codec/engine (`JSONEngine`/`JSONNode`, `URLCodec`, `Base64Codec`, `JWTCodec`) with its SwiftUI view. `JSONNode` is an `indirect enum`; numbers are stored as `String` to preserve precision; bools are told from numbers via `CFGetTypeID(number) == CFBooleanGetTypeID()`. Diagnostic logging flows through `AppLog` via the `.logErrors(_:message:)` view modifier (fires only on change, so it doesn't spam).

### Localization

- **`Strings`** (`Localization.swift`) is a **compile-checked** translation table — a `struct` whose fields are all filled by two static instances, `.vi` and `.en`, ported from the design handoff's `i18n.jsx`. A missing translation is a build error. Parameterized entries (e.g. `errLineCol`, `timeMin`, `jwtParts3`) are stored closures.
- **`LocalizationManager`** mirrors `ThemeManager`: `@AppStorage("devkit-lang")` (default **`en`**), `var lang: Lang`, `var s: Strings` (read in views as `loc.s.<field>`, paralleling `theme.t.<token>`), and `var locale: Locale` (applied via `.environment(\.locale, …)` in `RootView` so date/number formatting follows the language).
- **Conventions:** user-facing strings go through `loc.s`. Two things are deliberately **not** localized: `ToolID.name` (stable log category) and the `*.errorText` diagnostic feeds. Engine error messages we author ourselves (URL/Base64/JWT) take a `Strings` argument so they localize; JSON parser messages come from Foundation (already English) and only their line/col wrapper is localized. Technical labels (`JWT Token`, `Header`/`Payload`/`Signature`, `encodeURIComponent`, `URL-safe`) stay literal.

- **`Item.swift` / SwiftData** (`ModelContainer` in `XoaiUtilityApp`) are leftover template boilerplate — `Item` is not used by any view. Don't assume persistence is part of the feature set.
